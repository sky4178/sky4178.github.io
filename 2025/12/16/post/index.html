MyBatis多参数传递之@Param究竟加还是不加？

发表于2025-12-16|更新于2025-12-30
|总字数:542|阅读时长:2分钟|浏览量:

MyBatis 里的 @Param 常被“凭感觉”使用：有时候不写也能跑，有时候不写就报错。下面讲清楚：什么时候必须写、为什么有时能省、如何优雅收束。

什么时候必须写 @Param

  1. 多个参数：方法有两个及以上参数时写清名字，避免 MyBatis 自动生成的 param1/param2。
  2. 需要别名：想让 SQL 中的变量名更语义化时。
  3. SQL 用到了 ${}：当需要拼接列名或表名时，参数也要标记，否则无法命中。谨慎使用 ${}，注意注入风险。
  4. 动态 SQL 中引用参数：哪怕只有一个参数，只要在 <if/>、<choose/> 等里用到它，也最好显式声明。

速查清单

  • 只有一个参数，SQL 里也不做动态判断：可不写，但写了更稳。
  • 任何 ${}、动态 SQL、多个参数场景：必须写。
  • 想换个更语义化的名字：写。

为啥有时不写也能跑？

IDEA 或 Maven 若给编译器加了 -parameters，编译出的字节码会保留方法参数的真实名字。MyBatis 读取到这些名字后，就能用 deviceId、deviceName 代替默认的 param1/param2，于是“没写也能用”。

风险在于：

  • 换了构建方式、JDK 或关闭 -parameters，参数名丢失，瞬间报错。
  • 团队里有人没开这个编译选项，拉代码后就踩坑。

最佳实践

  • 多参数、动态 SQL、${} 场景一律加 @Param，不要赌编译参数。
  • 需要传入列名/表名时，白名单校验后再用 ${}，减少注入风险。
  • 项目统一在 pom.xml 或编译脚本里声明 -parameters，但把它当成兜底，不是省略 @Param 的理由。
