乐观锁和悲观锁

发表于2026-03-10|更新于2026-03-10
|总字数:677|阅读时长:2分钟|浏览量:

并发修改同一条数据时，最常见的问题就是“丢失更新”。这也是乐观锁和悲观锁存在的核心原因。

悲观锁是“先锁再改”，乐观锁是“提交时校验再改”。

比如账户余额是 100，两个请求同时扣款：

  • 请求1扣 20
  • 请求2扣 30

如果都先读到旧值 100，最后结果可能变成 70，而正确结果应该是 50。

为了解决这个问题，常见方案只有两类：

  • 悲观锁：先锁住，再修改
  • 乐观锁：先不锁，提交时检查数据有没有被别人改过

悲观锁

悲观锁假设冲突经常发生，所以直接把记录锁住。

常见写法：

拿到锁的事务先执行，其他事务要等它提交后才能继续。优点是结果稳定、控制强；缺点是会阻塞，吞吐量低，还有死锁风险。

适合场景：

  • 写多读少
  • 热点数据竞争激烈
  • 数据不能出错，且可以接受等待

乐观锁

乐观锁假设大多数时候不会冲突，所以不提前加锁，而是在更新时带上条件。

最常见的是 version 方案：

只有当数据库里的 version 还是旧值时，这次更新才会成功；如果别人已经改过，这条 SQL 会影响 0 行，业务层再决定是否重试。

它的关键在于：update ... where ... 是数据库原子操作，判断和更新一次完成，所以不会发生错误覆盖。

适合场景：

  • 读多写少
  • 冲突概率低
  • 可以接受失败后重试
  • 更关注并发性能

两者区别

维度 乐观锁 悲观锁
思路 提交时检查 修改前先加锁
是否阻塞 通常不阻塞 会阻塞
并发性能 高 较低
冲突处理 失败或重试 等待
常见实现 version、旧值判断 for update

实际怎么选

可以直接按这个原则判断：

  • 读多写少、普通 CRUD：优先乐观锁
  • 冲突频繁、热点竞争强：优先悲观锁
  • 库存扣减这类固定规则更新：优先原子条件更新

这不一定是标准 version 乐观锁，但本质也是并发控制，因为它把“校验”和“扣减”合成了一条原子 SQL。

一句话总结

  • 悲观锁：我先锁住，别人别动
  • 乐观锁：你们先改，提交时我再检查

如果你用的是 MyBatis-Plus，工程上通常更推荐直接用 @Version 做标准乐观锁。
