Dockerfile 是做什么的

Dockerfile 就是镜像的构建说明书，核心流程通常只有这几步：

  • 选基础镜像：FROM
  • 拷贝代码：COPY
  • 安装依赖：RUN
  • 暴露端口：EXPOSE
  • 指定启动命令：CMD

Docker Compose 是做什么的

如果项目不止一个容器，比如 web + mysql，一个个 docker run 会很乱，这时更适合用 docker compose 统一管理。

最常见的 compose.yml 结构包括：

  • services：定义有哪些服务
  • ports：端口映射
  • volumes：数据卷
  • environment：环境变量
  • depends_on：启动依赖

乐观锁和悲观锁
并发修改同一条数据时，最常见的问题就是“丢失更新”。这也是乐观锁和悲观锁存在的核心原因。 悲观锁是“先锁再改”，乐观锁是“提交时校验再改”。 比如账户余额是 100，两个请求同时扣款： 请求1扣 20 请求2扣 30 如果都先读到旧值 100，最后结果可能变成 70，而正确结果应该是 50。 为了解决这个问题，常见方案只有两类： 悲观锁：先锁住，再修改 乐观锁：先不锁，提交时检查数据有没有被别人改过 悲观锁悲观锁假设冲突经常发生，所以直接把记录锁住。 常见写法： 1234select balancefrom userwhere id = 1for update; 拿到锁的事务先执行，其他事务要等它提交后才能继续。优点是结果稳定、控制强；缺点是会阻塞，吞吐量低，还有死锁风险。 适合场景： 写多读少 热点数据竞争激烈 数据不能出错，且可以接受等待 乐观锁乐观锁假设大多数时候不会冲突，所以不提前加锁，而是在更新时带上条件。 最常见的是 version 方案： 123update userset balance = 80, version = 2where id ...
